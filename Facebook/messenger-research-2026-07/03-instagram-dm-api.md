# Instagram DM Automation via Meta APIs — Ground Truth (as of July 2026)

**Method note:** All claims below were pulled from live scrapes of Meta's official docs (July 12, 2026). Meta reorganized its docs IA in ~2026: `developers.facebook.com/docs/messenger-platform/instagram/...` now redirects to `developers.facebook.com/documentation/business-messaging/instagram-messaging/...`, and `docs/instagram-platform` → `documentation/instagram-platform`. Your existing bookmarks still resolve, but canonical URLs below are the new ones.

---

## 1. The two API generations — status and direction

**Both are alive, actively maintained, and officially parallel.** The Instagram Platform Overview (updated Jun 30, 2026) presents them as two co-equal "API configurations" ([documentation/instagram-platform/overview](https://developers.facebook.com/documentation/instagram-platform/overview)):

| | Messenger Platform "Instagram Messaging" (your current path) | Instagram API with Instagram Login |
|---|---|---|
| Auth | Facebook Login for Business; Facebook User/**Page access token** | Business Login for Instagram; **Instagram User token** |
| Host | `graph.facebook.com`, `POST /<PAGE_ID>/messages` (or `/me/messages`) | `graph.instagram.com`, `POST /<IG_ID>/messages` with `Authorization: Bearer` |
| Facebook Page | **Required** | **Not required** ([IG API w/ IG Login page](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login): "does not require a Facebook Page to be linked") |
| Permissions | `instagram_basic`, `instagram_manage_messages`, `pages_manage_metadata` (+ `pages_show_list`, `business_management`) | `instagram_business_basic`, `instagram_business_manage_messages` |
| Docs home | [documentation/business-messaging/instagram-messaging](https://developers.facebook.com/documentation/business-messaging/instagram-messaging) | [documentation/instagram-platform/instagram-api-with-instagram-login/messaging-api](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login/messaging-api) |

- **No sunset of the Page-linked path is announced anywhere.** Its docs are being actively updated (Send a Message updated Jul 2, 2026; Overview Jun 26, 2026; App Review Jun 21, 2026), and new features (PDF attachments Dec 2025, attachment-ID image sends Mar 2026) shipped to **both** stacks simultaneously ([changelog](https://developers.facebook.com/documentation/instagram-platform/changelog)).
- **Meta's steer is segmentation, not replacement:** every Page-linked doc now carries the note "If your app users don't have a Facebook Page linked to their Instagram professional account, learn more about the Instagram API with Instagram Login" ([Send a Message](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/features/send-message)). The [Migration Guide](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login/migration-guide) (Aug 21, 2024) pitches IG Login on onboarding friction ("from an average of 12 steps to just two") but frames migration as a *choice* driven by whether your users have Pages.
- **Feature deltas that matter for choosing:** IG-Login setup cannot access ads or tagging ([limitations](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login)); Product template, Hashtag Search, Product tagging, Partnership Ads are Facebook-Login-only ([overview comparison table](https://developers.facebook.com/documentation/instagram-platform/overview)). The Page-linked Send API also gets a **higher documented send rate** (300/sec vs 100/sec — see §8).
- **Coexistence on one app: yes.** The overview's "Develop with Meta" section documents adding *both* product sets to a single Business-type app — "**Instagram > API setup with Instagram login**" and "**Messenger**, including Instagram settings + **Instagram > API setup with Facebook login**" are just different products on the same App Dashboard ([overview](https://developers.facebook.com/documentation/instagram-platform/overview)). Caveats: an IG-Login integration uses a **separate Instagram App ID/secret** (found under Instagram > API setup with Instagram login), and IG-Login App Review is handled "within the Instagram product flow instead of the App Review item" ([migration guide](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login/migration-guide)). The webhooks fields table explicitly enumerates all three integration columns (Business Login for IG / Facebook Login / Instagram Messaging via Messenger Platform) against one app's subscriptions ([IG webhooks](https://developers.facebook.com/documentation/instagram-platform/webhooks)).
- **Multi-tenant recommendation (grounded in the above):** keep the Page-linked Messenger Platform path as primary (higher send throughput, product template, Marketing/notification messages, handover-successor Conversation Routing is documented against it), and add the IG-Login path as a **second onboarding lane** for tenants without Pages — Meta explicitly designed it for that population and both can live on one app. **Scope deprecation already bit here once:** the original IG-Login scopes (`business_manage_messages` etc.) were deprecated **Jan 27, 2025** in favor of `instagram_business_manage_messages` etc. ([IG API w/ IG Login](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login)).

## 3. IG-specific webhook events (current field list)

Object is `"object": "instagram"`, `entry.id` = IG account ID. Full field/permission matrix: [IG Platform Webhooks](https://developers.facebook.com/documentation/instagram-platform/webhooks); payload shapes: [Instagram Messaging Webhooks](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/webhooks) (updated Jun 26, 2026) and [webhook examples](https://developers.facebook.com/documentation/instagram-platform/webhooks/examples).

- **`messages`** — text, attachments (`audio|file|image|video|share|story_mention|ig_reel|reel`), `is_deleted: true` (customer deleted a message), `is_echo: true` (your account sent it — echoes are folded into `messages` on the Page-linked path; a **separate `message_echoes` field exists only on the IG-Login config**), `is_unsupported`, `quick_reply.payload`, `referral` (IG Shops product or CTD ad with `ads_context_data`), `reply_to.mid` (inline reply), `reply_to.story {url, id}` (story reply). **Story replies**: arrive as messages with `reply_to.story`; since **Dec 12, 2025** include `link_sticker_url` ([changelog](https://developers.facebook.com/documentation/instagram-platform/changelog)). **Story mentions**: `attachments type: story_mention` with CDN URL — you must not store/cache the media, only the URL; mention webhooks only flow from public accounts (or private accounts that follow you) ([Story Mention](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/features/story-mention)).
- **`message_reactions`** — `reaction {mid, action: react|unreact, reaction: love|..., emoji}`; v12+ supports `angry, sad, wow, love, like, laugh, other`. No echo webhook for your own reactions.
- **`message_edit`** — **NEW Sep 10, 2025**, both login configs: `message_edit {mid, text, num_edit}` ([changelog](https://developers.facebook.com/documentation/instagram-platform/changelog), [example payload](https://developers.facebook.com/documentation/instagram-platform/webhooks/examples)).
- **`messaging_postbacks`** — icebreaker/generic-template/persistent-menu button taps; `mid` included from v11+.
- **`messaging_seen`** — read receipts: `read {mid}`.
- **`messaging_referral`** — `ig.me` links with `?ref=` (existing conversations only, `type: OPEN_THREAD`).
- **`messaging_optins`** — IG-Login config only (per fields table).
- **`messaging_handover` / `standby`** — routing/thread-control events (see §4 note on Conversation Routing).
- **`response_feedback`** — Page-linked config only.
- Plus non-messaging IG fields: `comments`, `live_comments` (Advanced Access required), `mentions`, `story_insights`.
- **Payload gotchas (documented):** inbound gif/sticker messages fire **no webhook at all**; carousel reactions/forwards reference the first image; shares include only the URL; batches up to 1,000 events, retried with backoff for 36 hours ([Webhooks limitations](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/webhooks), [IG webhooks](https://developers.facebook.com/documentation/instagram-platform/webhooks)). A **post-share attachment-type change** is mid-transition since Oct 27, 2025 ([Instagram Post Shares: New Attachment Type and Transition Period](https://developers.facebook.com/documentation/instagram-platform/webhooks/new)).

## 4. Comment-to-DM (Private Replies)

Source: [Private Replies (Page-linked)](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/features/private-replies) (updated Jul 2, 2026) and [Private Replies (IG Login)](https://developers.facebook.com/documentation/instagram-platform/private-replies).

- Mechanics: on a `comments` or `live_comments` webhook, `POST /<PAGE_ID>/messages` (or `/<IG_ID>/messages` on IG Login) with `recipient: {comment_id: <COMMENT_ID>}`. Works for comments on posts, **ads posts, reels, and live**.
- **Windows: 7 days** from comment creation for posts/ads/reels; **Instagram Live comments only during the broadcast**.
- **One message only** per comment; the message auto-includes a link to the post. The conversation only continues (24-hour window opens) **if the user replies** to your private reply.
- Delivery folder: user's **Inbox if they follow you, Requests if not**.
- Permissions (Page-linked): `instagram_manage_comments` + `pages_messaging`, Human Agent feature, **Advanced Access** required. Rate limit: 750/hr per IG account for post/reel comments; 100/sec for live (§8).
- Ads-post comments include `ad_id`/`ad_title` and may produce duplicate webhooks.

**Related change:** the **Handover Protocol is no longer supported for Instagram** — "All businesses have been migrated to Conversation Routing" (backwards-compatible with most Handover APIs; thread control default 24h, extendable to 7 days via API, and up to 30 days from ads via `receiving_app_control_expiration`) — [Conversation Routing for Instagram](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/features/conversation-routing) (updated Jun 30, 2026). If your channel design assumed Handover, target Conversation Routing.

## 5. Eligibility & account requirements

- **Professional account (business or creator) required** — "Instagram Messaging is available for... Any Instagram Professional account for a business [or] Creator" ([Overview](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/overview)).
- **No follower-count gates and no country restrictions appear in the current docs** — the old 2021-era follower caps and country rollout language are gone from all pages scraped.
- **Page linkage: still required in 2026 for the Messenger Platform path** ("a Facebook Page connected to that account", [Getting Started](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/get-started), updated Apr 1, 2026); **not required** on the IG-Login path.
- **The "Allow access to messages" toggle is still mandatory**: "Instagram Settings > Messages and story replies > Message controls > Connected Tools > toggle **Allow Access to Messages**" ([Getting Started §4](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/get-started)).
- Page-linked path token chain: user must hold the `MESSAGE`/`MODERATE` task on the Page (`PROFILE_PLUS_MESSAGING` grants `instagram_manage_messages` — [overview task table](https://developers.facebook.com/documentation/instagram-platform/overview)); long-lived user token → non-expiring Page token. A new **Instagram setup tool** (App Dashboard → Messenger → Instagram Settings) is rolling out to generate Page tokens + webhooks directly ([Getting Started §3a](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/get-started)).
- **Creator-account quirk:** a Creator account "must send an API request using the Instagram Conversations API before the account can receive webhooks" ([Overview limitations](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/overview)).
- IG-Login path tokens: authorization code (1h) → short-lived (1h) → long-lived (60 days, refreshable) Instagram User token ([overview](https://developers.facebook.com/documentation/instagram-platform/overview)).

## 6. Message Requests folder mechanics

Documented in both [Overview §Instagram inbox](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/overview) and [IGL Send Messages](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login/messaging-api):

- **Followers → Primary; non-followers → Requests.** Pre-existing conversations stay where the account had filed them.
- Request messages **aren't marked Seen until accepted**; user can accept/deny in-app.
- **Replying via a third-party app moves the conversation to the General folder — always**, regardless of inbox settings. (i.e., your bot's reply effectively accepts the request; the docs describe answering requests via API as normal flow, so automation does receive and can reply to Requests-folder messages. There is no documented webhook suppression for Requests.)
- **Folder info is not exposed to the API**: "Inbox folders are not supported and messages delivered by the Messenger Platform do not include the folder information."
- **API delivery ≠ Read**: webhook/API delivery never marks a message Read in the IG app; only your reply does.
- **30-day expiry**: "Conversations in the Requests folder that have not been active for 30 days will not be returned in API calls."
- Private replies land in the recipient's Inbox (follower) or Requests (non-follower) — see §4.

## 7. Human Agent tag on Instagram

**Available on both paths.** The **Human Agent feature** (App-Review-gated, listed for both Instagram-login and Facebook-login permission sets) "allows your app to have a human agent respond to user messages using the **`human_agent`** tag **within 7 days** of a user's message," for cases the 24-hour standard window can't cover (weekend closures, >24h investigations) ([Instagram Platform Overview §Features and permissions](https://developers.facebook.com/documentation/instagram-platform/overview), updated Jun 30, 2026; echoed in §Messaging: "you can use the human agent tag to send a response within 7 days"). The Page-linked overview likewise: "If a human agent needs more time to respond, you can add a tag to your response to send the message outside the standard 24-hour messaging window" ([Overview](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/overview)). Apps using IG messaging **must have a human-agent escalation path** as a review requirement — single-app custom inbox or multi-app via Conversation Routing ([Human Agent Escalation](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/features/human-agent-escalation)).

## 8. Rate limits (IG messaging-specific, per-account not conversation-based)

Source: [Graph API Rate Limiting](https://developers.facebook.com/docs/graph-api/overview/rate-limiting) — IG messaging is carved out of the impressions-based Instagram BUC formula and counted **per IG professional account, per API**:

**Messenger API for Instagram (Page-linked, our path):**
- Send API: **300 calls/sec/IG account** for text, links, reactions, stickers; **10 calls/sec** for audio/video.
- "Your app may be rate limited if too many messages are being sent to a single thread."
- Conversations API: **2 calls/sec/IG account**.
- Private Replies: **100 calls/sec** (Live comments), **750 calls/hour** (post/reel comments).

**Instagram API with Instagram Login:** same shape but Send API is **100 calls/sec** (text/links/reactions/stickers), 10/sec audio-video; Conversations 2/sec; Private replies 100/sec live, 750/hr posts ([also in IG Platform Overview §Rate Limiting](https://developers.facebook.com/documentation/instagram-platform/overview)).

Non-messaging IG endpoints use the BUC formula `calls/24h = 4800 × impressions` per app-user pair. Watch the `X-Business-Use-Case-Usage` header (`type: instagram`, error code 80002).

## 9. App Review, access levels, dev-mode, testers

- **Access levels:** Standard Access (default) = your app "can only interact with people who have a role on the app"; **Advanced Access** = required to serve accounts you don't own/manage; requires **App Review + Business Verification** ([IG Platform Overview §Access levels](https://developers.facebook.com/documentation/instagram-platform/overview); [Instagram Messaging Overview §Testing limitations](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/overview): "Some features may not work properly until your app has been granted Advanced Access").
- **`instagram_manage_messages` review (Page-linked):** submit via App Dashboard → App Review → Requests; must show webhook processing, Send API + (if custom inbox) Conversations API usage end-to-end; separate guide tracks for "Apps for Other Businesses" vs "Apps for Your Own Business"; Page must be connected to the IG account at review time ([App Review for Messenger API support for Instagram](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/app-review), updated Jun 21, 2026). Human-agent escalation must be demonstrable (§7). On the IG-Login side, review happens **inside the Instagram product flow** in the dashboard, not the classic App Review panel ([migration guide](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login/migration-guide)).
- **"App must be LIVE for webhooks": CONFIRMED, documented in two places.** "Your app must be published, regardless of app review status, to receive webhooks. ... If your app has not been approved... Webhooks will only be sent if the person using your app has a role on the app" ([Instagram Messaging Webhooks §Requirements](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/webhooks)); "Apps must be set to **Live** in the App Dashboard to receive webhook notifications" ([IG Platform Webhooks §Limitations](https://developers.facebook.com/documentation/instagram-platform/webhooks)). So: Live app + role-holder = webhooks under Standard Access; Development mode = no IG webhooks, period.
- **Testers:** "Your app testers must have a role on your app, grant your app access to all the required permissions, **and have a role on the Instagram professional account** that owns the app" ([IGL Send Messages §Limitations](https://developers.facebook.com/documentation/instagram-platform/instagram-api-with-instagram-login/messaging-api)). The account-level webhook subscription must also be enabled per account via `POST /me/subscribed_apps?subscribed_fields=...` ([IG webhooks §Enable Subscriptions](https://developers.facebook.com/documentation/instagram-platform/webhooks)).

### Watch-outs for our current implementation specifically
1. Our exact stack (Page token, `graph.facebook.com/.../me/messages`, `entry.id` = IG ID) is the documented, current shape — no forced migration.
2. Subscribe to `message_edit` (Sept 2025) and handle `is_deleted` — Meta Platform Terms require deleting stored copies on deletion webhooks ([Overview §Requirements](https://developers.facebook.com/documentation/business-messaging/instagram-messaging/overview)).
3. If we rely on Handover Protocol semantics anywhere, re-validate under Conversation Routing.
4. The Oct 2025 post-share attachment-type transition may change payloads we already parse.
5. Persistent menu quirks: no composer lock, menus don't live-update, and a menu tap opens a fresh 24h window — that's a free re-engagement surface IG *does* support, contrary to common belief.
