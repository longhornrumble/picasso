### §3.1 — M2 done-bar #2 wording correction

**Plan v0.27 wording (master plan §M2 done-bar #2):**
> tenant-scoped Scan + FilterExpression against `picasso-channel-mappings-staging` reaches every PSID-keyed subject

**Actual M2 walker design:** `picasso-channel-mappings-staging` maps `PAGE#{pageId} → tenantId` only. **PSIDs are never written to channel-mappings** — they appear only as suffixes on `sessionId="meta:{pageId}:{psid}"` rows in `staging-recent-messages` (and any other Meta-session-keyed table).

**Correct design:** subject resolution for `identifier_type=psid` proceeds in two steps:

1. **Tenant → pageIds:** Query channel-mappings `TenantIndex` GSI (KEY: `tenantId=HASH, channelType=RANGE`) with `tenantId={input}, channelType=messenger` → returns list of `PK="PAGE#{pageId}"` rows. Extract pageIds.
2. **(pageId, psid) → sessionIds:** for each pageId, construct `sessionId = f"meta:{pageId}:{psid}"`. Pass the resulting list to the existing `_walk_recent_messages` walker (M1; line 849), which Queries `staging-recent-messages` by sessionId.

**Master plan update target:** Sprint E (closure PR) edits M2 done-bar #2 to read:
> tenant-scoped Query on `picasso-channel-mappings-staging.TenantIndex` GSI returns pageIds; constructed `sessionId="meta:{pageId}:{psid}"` reaches every PSID-keyed subject via `_walk_recent_messages` (reused). Integration test populates ≥2 pages per tenant; asserts walker output covers all synthetic Meta sessions with zero false-negatives.

### §3.2 — Additional Meta-session surfaces beyond `staging-recent-messages`

`picasso-session-summaries-staging` (`pk=TENANT#{tenant_hash}, sk=SESSION#{sessionId}`) and `picasso-session-events-staging` (`pk=SESSION#{sessionId}, sk=STEP#{n}`) may carry Meta-session content (sessionId starts with `meta:` prefix). Both surfaces are currently **DEFERRED** in `lambda_function.py:135-147` (`conversation-summaries` entry) for M1 reasons, but the same `meta:{pageId}:{psid}` pattern reaches them.

**Sprint A decision:** route session-summaries + session-events walker work as **M2 done-bar #1 sub-items**, not as separate M2 done-bar items. The two surfaces share the same subject-resolution input (psid → sessionIds) as `_walk_recent_messages`; adding them is a per-surface walker function on the same input list.

**Trigger to re-evaluate:** if Sprint B integration test finds Meta sessions in session-summaries OR session-events that don't appear in recent-messages (TTL drift OR session-archiver behavior), the gap escalates.

### §3.3 — `identifier_type` policy for `phone` and `name+address`

**Master plan done-bar #5 directive:** implement walker support OR record explicit walker-not-supported + manual procedure in M3 playbook + named D5 row.

**Sprint A decision:** **walker-not-supported** for both `phone` and `name+address`. Rationales:

- **`phone`:** the platform does NOT persist phone numbers as primary subject identifiers on any tenant-scoped row. SMS_Sender (`SMS_Sender/`) writes `picasso-notification-sends-staging` keyed by `pii_subject_id` (already covered by M1 walker via subject-resolution). SMS consent (`picasso-sms-consent-staging`) is keyed differently and is an opt-in artifact, not subject-rights-actionable. Phone-only subjects can be served via the M3 manual playbook procedure: operator runs `aws dynamodb scan --filter-expression "phone = :p"` on `picasso-sms-consent-staging` (per-tenant; bounded item count).
- **`name+address`:** name/address fields are free-text within form responses (`picasso-form-submissions-staging.responses` map); they have no index or column projection. The M1 walker already exports them as part of the form-submissions row dump for email-resolved subjects. Subjects who provided name+address but no email cannot be matched without manual review. Operator playbook covers via `aws dynamodb scan` snippet over form-submissions with FilterExpression on `contains(responses, :name)`.

**Routing:** new D5 row **F-DSAR30** = "phone / name+address `identifier_type` not walker-supported; manual playbook procedures cover the gap." Severity: L/L (subject populations are small; manual procedure is documented). Open with named trigger = first DSAR request with `identifier_type ∈ {phone, name+address}` in production.

**M3 playbook update:** Sprint E adds two §"Per-surface manual fallback procedures" entries (one for each unsupported identifier_type) to `dsar-operator-playbook.md`.

## §5 — Subject-resolution model

The Lambda's `_resolve_subject(tenant_id, normalized_email)` currently maps `email → pii_subject_id` via `picasso-pii-subject-index-staging` (M1). M2 introduces a parallel resolver for `psid`.

**Pattern:**

```python
def _resolve_psid_subject(tenant_id, psid):
    """Resolve psid → list of sessionIds for the M2 walker.

    Returns: list[str] of sessionIds suitable for _walk_recent_messages /
    _walk_session_summaries / _walk_session_events / S7 archive walker.

    Note: there is no opaque pii_subject_id assigned to Meta-only subjects
    today. The walker operates directly on (psid, pageIds) — psid IS the
    subject identifier for Meta-only data, scoped within tenant via pageIds.
    """
    # Step 1: tenant → pageIds via TenantIndex GSI
    response = ddb.Table("picasso-channel-mappings-staging").query(
        IndexName="TenantIndex",
        KeyConditionExpression="tenantId = :t AND channelType = :c",
        ExpressionAttributeValues={":t": tenant_id, ":c": "messenger"},
        ProjectionExpression="PK",
    )
    page_ids = [item["PK"].split("#", 1)[1] for item in response["Items"]
                if item["PK"].startswith("PAGE#")]
    # Step 2: build sessionIds
    return [f"meta:{page_id}:{psid}" for page_id in page_ids]
```

**Properties:**
- Tenant-scoped (Query on GSI bounded by tenantId; cross-tenant isolation preserved).
- No false-positives (every sessionId begins with `meta:` so it's distinguishable from widget sessions in the same table).
- False-negatives bounded by channel-mappings completeness — every Meta pageId for the tenant is enumerated (Sprint B integration test asserts this).

## §6 — ARCHIVE-decision M2 validation

The Phase 0.5 deliverable `archive-reachability-decision.md` (2026-05-23) is **the source of truth** for the archive walker's design. M2 Sprint C does NOT re-do the decision; it cites it and implements the walker.

**Key inputs from that doc (do not re-derive in Sprint C):**

- Bucket name: `picasso-archive-staging` (staging-only — prod-614 has no archive surface; F-DSAR17 routes to M9 TTL hygiene)
- Region: `us-east-1`
- Encryption: SSE-S3 (AES256; no CMK)
- Versioning: **ENABLED** with 7-day `NoncurrentVersionExpiration` compensating control
- Lifecycle: `DeleteAfterOneYear` on `sessions/` prefix
- Key shape: `s3://picasso-archive-staging/sessions/{sessionId}/...`
- Decision: **walk** the archive bucket; version-aware delete required because versioning is ON

**Sprint C walker contract:**

```python
def _walk_archive_bucket(session_ids, request_type, dry_run):
    """Walk picasso-archive-staging for given sessionIds.

    For request_type='access': list versions + return object metadata.
    For request_type='delete' + dry_run: count objects + versions; no delete.
    For request_type='delete' + real: enumerate versions via
    list_object_versions, then delete_object per (key, version_id).

    Continue-on-error per session_id (mirrors M1 walker pattern).
    """
```
